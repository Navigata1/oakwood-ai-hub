Display Control System

The new integration built Apr 13–14
Root Cause Discovery
BED-H is a Pro TV, not a signage display
The Samsung LH98BEDHLGFXGO (and 75" counterpart LH75BEDHLGFXGO) are branded as commercial "Business TVs" but run Samsung Tizen 9.0 and use the consumer Smart TV API stack, not the binary MDC protocol used by QM/QH/DB signage displays. That's why Samsung's MDC Unified software connects (TCP handshake succeeds on port 1516) but reports zero detected devices — the displays aren't speaking the MDC dialect. Solution: control them the same way a Samsung Smart View / SmartThings client does — via the WebSocket KEY_POWER endpoint on port 8002 with token-based pairing.

Control Flow — Power ON

Step 1Wake-on-LAN UDP:9
Step 2Wait 3s
Step 3EXLINK TCP 1516
Step 4WS KEY_POWER (if off)
Step 5Verify via HTTP 8001

Control Flow — Power OFF

Step 1Query HTTP 8001
Step 2If on → WS KEY_POWER
Step 3EXLINK TCP 1516
Step 4Verify via HTTP 8001

Pairing Sequence

Step 1Open wss 8002
Step 2Send device name
Step 3Await connect event
Step 4Capture token
Step 5Persist JSON
The One Setting That Made Everything Work
Access Notification → Always Allow
On Tizen 9 firmware 2115, the Device Connect Manager moved from General → External Device Manager to All Settings → Connection → Device Manager. The key setting is Access Notification — if it's anything other than Always Allow, unknown devices get silently rejected with ms.channel.timeOut and no pairing popup is ever shown. Setting it to Always Allow on all 5 displays was the unlock.

Protocol Stack

Multi-layer control reference
Primary — Samsung Smart TV WebSocket
NUC Controller— wss 8002 —>/api/v2/channels/samsung.remote.control— KEY_POWER —>TV Tizen
Token-based auth, persistent per TV. First connection triggers "Access Notification"
popup on the TV — one-time approval, then token is saved for all future calls.
Fallback — EXLINK on TCP 1516
NUC Controller— TCP 1516 —>Raw bytes— 7-byte command —>TV legacy daemon
# Power ON
0x08 0x22 0x00 0x00 0x00 0x02 0xD4

# Power OFF
0x08 0x22 0x00 0x00 0x00 0x01 0xD5
Older Samsung control protocol. Works regardless of pairing. Useful when TV is in a state
where WebSocket is unresponsive. No ACK from TV — fire and forget.
Wake — Wake-on-LAN
NUC Controller— UDP 9 —>Magic packet— 6× FF + 16× MAC —>TV NIC
Sent to broadcast 10.1.240.255 AND direct IP. Wakes the TV from deep standby
so subsequent EXLINK / WS commands can land.
Status — Samsung Smart TV HTTP API
NUC Controller— HTTP GET —>http://[IP]:8001/api/v2/— JSON —>device.PowerState
{
 "device": {
 "modelName": "LH98BEDHLGFXGO",
 "PowerState": "on",
 "networkType": "wired",
 "TokenAuthSupport": "true",
 "OS": "Tizen"
 }

HDCP Issue — ZV Pro Broadcast Blocker

Carried from v2 — still unresolved
Problem

ZeeVee ZV Pro at 10.1.240.4 reports "No Video" because the HDMI source is HDCP-encrypted. The ZV Pro can't pass HDCP content. Verified: same HDMI signal displays correctly on a standard monitor.

Proven Fix (from Jimmy's church install)
Source→ HDMI →HDCP-Stripping Splitter→ Clean HDMI →ZV Pro

Order 2–3 powered 1x2 HDMI splitters: ViewHD VHD-1X2MN3D (~$18), OREI HD-102 (~$15), HDFury Integral 2 (premium).
